# 📚 โครงสร้างระบบย่อลิงก์ (URL Shortener) - Guide สำหรับมือใหม่

โปรเจกต์นี้เป็นตัวอย่างที่ดีของการทำ **Web Application แบบ SPA (Single Page Application) เบื้องต้น** โดยใช้เครื่องมือที่ทันสมัยแต่ไม่ซับซ้อนจนเกินไป

## 1. ส่วนแสดงผล (Frontend)
ส่วนนี้คือสิ่งที่ผู้ใช้มองเห็นและโต้ตอบด้วยบนเบราว์เซอร์

*   **`index.html` (โครงกระดูก):**
    *   ทำหน้าที่เป็นหน้ากระดาษเปล่าที่กำหนดว่าบนเว็บมีอะไรบ้าง (เช่น ช่องข้อความ, ปุ่มกด)
    *   **💡 ข้อแนะนำ:** สังเกตการใช้ `<div id="app">` และ `<div id="redirecting">` เราใช้เทคนิคการซ่อน/แสดง `div` เหล่านี้ด้วย CSS (คลาส `.hidden`) เพื่อสลับหน้าจอไปมาโดยที่**ไม่ต้องโหลดหน้าเว็บใหม่เลย**
*   **`style.css` (ความสวยงาม):**
    *   ทำหน้าที่ตกแต่งหน้าตา (Theme สีดำสไตล์ Google)
    *   **💡 ข้อแนะนำ:** การใช้ตัวแปร CSS (เช่น `--bg-color: #202124;`) เป็นสิ่งที่ดีมาก เพราะถ้าอนาคตอยากทำ "ธีมสว่าง" (Light Mode) ก็แค่เปลี่ยนสีตัวแปรบรรทัดเดียว สีก็จะเปลี่ยนทั้งเว็บ
*   **`app.js` (สมองกล):**
    *   เป็นพระเอกหลักในการคิดคำนวณ ตรวจสอบ URL, สุ่มรหัสย่อ, และคุยกับฐานข้อมูล
    *   **💡 ข้อแนะนำ:** โค้ดในนี้ใช้เทคโนโลยี `async / await` ซึ่งจำเป็นมากเวลาทำเว็บที่ต้องคุยกับ Database เพราะต้อง "รอ" ให้ Database ตอบกลับมาก่อนถึงจะทำงานขั้นต่อไปได้

## 2. เครื่องมือช่วยพัฒนา (Build Tool)
ในการเขียนเว็บยุคใหม่ เรามักจะไม่เขียน HTML/JS เพียวๆ แล้วอัปโหลดเลย แต่จะมีเครื่องมือมาช่วยจัดการ

*   **Vite (`package.json` / `node_modules`):**
    *   เป็นเหมือนพ่อบ้านที่คอยเซิร์ฟเว็บให้เราดูตอนกำลังเขียนโค้ด (`npm run dev`)
    *   ช่วยให้เวลาเราเซฟไฟล์ปุ๊บ หน้าเว็บเบราว์เซอร์จะอัปเดตเองปั๊บ (Hot Reload)
    *   **💡 ข้อควรระวัง:** โฟลเดอร์ `node_modules` จะใหญ่มาก **ห้าม** อัปโหลดขึ้น GitHub เด็ดขาด (เราจึงต้องมีไฟล์ `.gitignore` มาคอยห้ามไว้)

## 3. ส่วนจัดการข้อมูล (Backend / Database)
เราไม่ได้เขียน Backend เอง แต่ใช้บริการสำเร็จรูป (BaaS - Backend as a Service)

*   **Supabase (ฐานข้อมูล PostgreSQL):**
    *   ทำหน้าที่จำว่า "รหัสย่อ ABC" คู่กับ "ลิงก์เว็บไซต์เต็มๆ อะไร"
    *   มีระบบ RLS (Row Level Security) เพื่อป้องกันไม่ให้คนอื่นมาดึงข้อมูลแปลกๆ ไปจากฐานข้อมูลของเรา
*   **`.env` (ตู้เซฟเก็บกุญแจ):**
    *   ใช้เก็บ `URL` และ `Anon Key` ของ Supabase
    *   **💡 ข้อควรระวังระดับสูงสุด:** ไฟล์ `.env` คือความลับสุดยอด ห้ามหลุดออกสู่สาธารณะเด็ดขาด (เราจึงต้องตั้งค่า `.gitignore` ไม่ให้อัปโหลดไฟล์นี้ขึ้น GitHub)

## 4. การเผยแพร่ (Deployment)
เมื่อเขียนโค้ดในเครื่องเสร็จ ก็ต้องเอาไปวางบน Server จริงให้คนทั้งโลกกดเข้ามาได้

*   **GitHub:** เป็นเหมือน Google Drive สำหรับโปรแกรมเมอร์ ใช้เก็บประวัติโค้ด
*   **Vercel:** เป็นคนทำหน้าที่นำโค้ดจาก GitHub มาเปิดเป็นเว็บไซต์ 24 ชั่วโมง
    *   **💡 ข้อแนะนำ:** การทำคู่กันแบบนี้เรียกว่า **CI/CD** (เขียนโค้ดลงเครื่อง -> ดันขึ้น Github -> Vercel ดึงไปอัปเดตเว็บให้อัตโนมัติ) ซึ่งเป็นมาตรฐานที่บริษัท Tech ระดับโลกใช้งานกันครับ

### 🚀 สรุปขั้นตอนการทำงานของระบบ (Flow)

**กรณีที่ 1: ผู้ใช้คลิกลิงก์ย่อ (เช่น `เว็บเรา.com/?c=ABcde`)**
1. โปรแกรมอ่าน `?c=ABcde` จาก URL
2. ไปค้นหาคำว่า `ABcde` ในตาราง Supabase
3. ถ้าเจอ: สั่งให้เบราว์เซอร์เปลี่ยนหน้าต่าง (Redirect) โยนไปหาเว็บเป้าหมาย
4. ถ้าไม่เจอ: แสดงข้อความ "ไม่พบลิงก์"

**กรณีที่ 2: ผู้ใช้เข้ามาสร้างลิงก์ย่อ**
1. ผู้ใช้กรอก URL เต็มๆ -> กดปุ่ม
2. เช็คก่อนว่า URL นี้เคยมีในฐานข้อมูลไหม? (ถ้ามี: ดึงรหัสเก่ามาตอบกลับเลย)
3. ถ้าไม่มี: ทำการสุ่มข้อความ 6 ตัวอักษร
4. บันทึกข้อความสุ่ม + URL เต็ม ลงในฐานข้อมูล Supabase
5. โชว์ลิงก์ที่ย่อสำเร็จ ให้ผู้ใช้กดคัดลอก (Copy)
